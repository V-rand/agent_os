此项目的目的是设计一个agentos系统为上层的律师伙伴工作流或者deep research服务，这个底层os需要有很强的上层拓展性，底层定义agent loop和提供稳定的tool、skills、memory支持，上层进行工作流等设计
所以这个项目主要是实现底层架构，不涉及具体的业务
github 仓库是git@github.com:V-rand/agent_os.git

学习它的（工具管理，skills管理，session管理，上下文管理，kv cache的控制，SQLite的使用，之前的demo也有使用SQLite，但是这记录下来的东西没啥作用，还是要像Hermes那样记录message才有用,上下文压缩机制，memory的接口，todo list stage的设计）
尽量复用Hermes工程代码，这可以缩短开发时间并且保障代码质量，注意需要删除我们这个项目不需要的，因为Hermes为了多设备多消息同时传入的gateway做了复杂的工程实现和防御性工程，而我们这个项目只有一个测试用的cli接口和一个网页端，不需要复杂的gateway

我对最终成品的想法，它是一个底层的agentos系统，不涉及业务，但是为上层的业务提供必要的支持
## 律师的工作
我们应该先别谈 agent、memory、workflow engine，先把**律师真实怎么工作**对齐。否则系统设计一定会偏成“AI 自己幻想的办案流程”。
我先用一个更朴素的视角来拆：律师的工作本质不是“按流程执行”，而是反复在做这几件事：

```text
接收信息
→ 理解事实
→ 判断法律关系
→ 判断风险和目标
→ 决定策略
→ 补充材料
→ 形成文书/意见/沟通
→ 推动外部程序
→ 根据新情况不断修正
```

律师真实工作流更像一个**持续迭代的案件判断过程**，而不是一条固定流水线。

## 1. 律师工作的第一步不是“立案”，而是“搞清楚到底发生了什么”

真实场景里，客户一开始给的信息通常是混乱的：

客户会说：

> “对方欠我钱不还。”
> “合同是签了，但他们没履行。”
> “我们被员工告了。”
> “这个项目现在很麻烦，你帮我看看。”

律师第一反应通常不是马上检索法律，也不是马上写文书，而是先做：

```text
这是什么事？
谁和谁之间的事？
时间线是什么？
有哪些文件？
客户真正想要什么？
现在最紧急的问题是什么？
```

所以第一阶段更准确地说是：

**事实接收与问题澄清。**

这个阶段律师的动作包括：

* 听客户描述；
* 看已有材料；
* 把杂乱事实整理成时间线；
* 识别关键主体；
* 识别关键文件；
* 追问缺失信息；
* 判断这到底是合同、劳动、侵权、婚姻、刑事、行政，还是混合问题；
* 判断是否有紧急期限或保全风险。

这里的核心产出不是“法律意见”，而是：

```text
初步事实框架
初步问题清单
材料缺口清单
紧急风险判断
客户目标理解
```

## 2. 第二步才是“把事实翻译成法律问题”

律师不会直接处理生活事实。律师会把生活事实转成法律结构。

比如客户说：

> “对方收了钱但没发货。”

律师脑子里会转换成：

```text
是否存在买卖合同关系？
合同是否成立并生效？
付款义务是否已履行？
交付义务是否到期？
对方是否构成违约？
能否解除合同？
能否主张返还价款、违约金、损害赔偿？
有没有管辖、时效、证据问题？
```

所以第二阶段是：

**法律关系识别与请求权/抗辩结构搭建。**

这一步律师在做的不是“查法条”，而是搭结构：

```text
事实 → 法律关系 → 请求权基础 → 构成要件 → 证据要求 → 风险点
```

例如：

```text
客户目标：要回货款
法律路径：
  1. 继续履行
  2. 解除合同 + 返还货款
  3. 违约损害赔偿
  4. 不当得利，若合同效力有问题

每条路径都要问：
  - 要件是什么？
  - 证据够不够？
  - 风险是什么？
  - 成本和周期如何？
```

## 3. 第三步是“判断客户目标是否现实”

律师真实工作中很重要的一步，是把客户的愿望翻译成可执行目标。

客户可能说：

> “我要让对方坐牢。”
> “我要他赔一千万。”
> “我要马上冻结他的账户。”
> “我要把他公司搞倒。”

律师要判断：

```text
这个目标法律上能不能实现？
证据上支不支持？
程序上来不来得及？
经济上值不值得？
有没有更优路径？
是否有谈判空间？
```

所以这一阶段不是单纯法律分析，而是：

**目标校准与策略选择。**

律师会形成几个方案：

```text
强硬诉讼方案
保守诉讼方案
先谈判后诉讼方案
先保全再起诉方案
发函施压方案
刑民交叉方案
行政投诉/举报方案
不建议继续推进方案
```

律师不是只回答“能不能赢”，而是帮助客户决定：

```text
值不值得打？
怎么打？
先做什么？
底线是什么？
风险是什么？
```

## 4. 第四步是“补材料”，而不是马上产出

真实律师工作里，很多案件不是分析不出来，而是材料不够。

律师经常会进入一个循环：

```text
看材料
→ 发现缺口
→ 问客户
→ 客户补材料
→ 再看
→ 再发现新问题
→ 再补
```

这其实是律师工作最常见的节奏。

材料缺口通常包括：

```text
合同有没有？
原件有没有？
付款凭证有没有？
聊天记录完整吗？
发票有没有？
送货单有没有？
对账单有没有？
对方主体信息有没有？
授权委托有没有？
关键时间点有没有证据？
```

所以真实流程不是：

```text
接案 → 分析 → 起诉
```

而更像：

```text
接案 → 初步分析 → 缺材料 → 补材料 → 修正分析 → 再补材料 → 才能决定怎么做
```

这一点对 agent 系统非常关键：律师工作台必须承认**案件早期是不完整、不确定、反复修正的**。

## 5. 第五步才是“形成专业产物”

律师最终会输出一些东西，但这些产物不是凭空生成的，而是前面判断的结果。

常见产物包括：

```text
法律咨询答复
案件分析意见
风险提示
证据清单
材料补充清单
律师函
起诉状
答辩状
代理意见
证据目录
庭审提纲
谈判方案
客户沟通纪要
```

这些产物大致可以分三类：

### 对客户的产物

比如：

```text
法律意见
风险说明
诉讼策略
费用和周期预估
下一步建议
```

### 对程序的产物

比如：

```text
起诉状
答辩状
证据目录
保全申请
管辖异议
上诉状
执行申请
```

### 对律师自己的产物

比如：

```text
事实时间线
争议焦点
证据审查表
庭审发问提纲
待办清单
案件备忘录
```

很多 AI 系统容易只盯着“生成起诉状”，但真实律师工作里，更有价值的往往是中间产物：

```text
事实表
时间线
证据缺口
争点表
策略备忘录
待办清单
```

## 6. 第六步是“推动外部流程”

律师不是只分析，还要推动事情发生。

比如诉讼律师会做：

```text
立案
缴费
保全
送达
举证
开庭
调解
判决
上诉
执行
```

非诉律师会做：

```text
尽调
访谈
审文件
改合同
出意见
谈判
交割
归档
```

顾问律师会做：

```text
日常咨询
合同审查
合规建议
劳动人事处理
函件往来
内部会议
专项项目
```

这里的关键是：律师工作不是一个单一流程，而是按业务类型不同而不同。

但它们都有共性：

```text
有事实输入
有法律判断
有材料管理
有对外沟通
有期限要求
有阶段推进
有结果沉淀
```

## 7. 第七步是“不断根据新情况修正”

这可能是最容易被忽略的地方。

律师不是一次性分析完就结束。真实案件会不断变化：

```text
客户又发来新证据
对方提交了答辩
法院要求补正
出现新的聊天记录
对方提出调解
客户改变目标
证据被法院不采信
法官关注点发生变化
一审输了，要不要上诉
执行中发现对方没财产
```

每出现一个新情况，律师都会重新判断：

```text
原来的事实判断还成立吗？
原来的法律路径还成立吗？
原来的诉讼请求要不要调整？
原来的证据策略要不要变化？
原来的风险等级有没有变化？
下一步动作是什么？
```

所以律师真实工作流不是线性的，而是一个循环：

```text
事实更新
→ 法律判断更新
→ 策略更新
→ 产物更新
→ 行动更新
→ 再进入下一轮
```

## 8. 所以律师真实工作流可以抽象成这个循环

我觉得更准确的抽象是：

```text
1. Intake：接收问题和材料
2. Sensemaking：整理事实，理解发生了什么
3. Legal Framing：转化为法律关系和法律问题
4. Goal Alignment：确认客户目标和现实约束
5. Gap Analysis：识别材料、证据、信息缺口
6. Strategy：选择路径和优先级
7. Production：形成文书、意见、清单、沟通内容
8. Execution：推动程序或商务动作
9. Monitoring：跟踪期限、反馈、新材料、新事件
10. Revision：持续修正事实、策略和产物
```

它不是：

```text
阶段 A → 阶段 B → 阶段 C → 阶段 D
```

而是：

```text
案件状态不断变化，律师不断重新理解、判断、推进。
```

## 9. 对齐到一句话

律师真实的工作不是“完成一个法律流程”，而是：

> 在信息不完整、目标会变化、外部程序不断推进的情况下，持续把事实、法律、证据、策略和行动对齐。

所以如果后面再谈 agent 设计，起点应该不是“agent 有哪些模块”，而是：

```text
这个 agent 如何陪律师完成：
理解事实
识别问题
补齐材料
判断风险
形成策略
产出文书
推进程序
持续修正
```

而不是一开始就问：

```text
怎么做记忆？
怎么做 workflow？
怎么做多 agent？
```

这些都应该服务于上面这个真实工作循环。

后续还会有一个python写的非agent的案件流程状态机更新目前案件的状态，我们的agent可以体现用户推进流程，但是流程控制要在agent之外，不然agent要处理的事情太多了而且这些事情太过于冗余。
agent主要是给用户提供高价值的建议，系统提供长期的案件管理支持
